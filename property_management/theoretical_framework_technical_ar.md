# الإطار النظري التقني وربطه بنماذج Odoo

## 1. الهدف من هذه الوثيقة

هذه الوثيقة تربط المفاهيم النظرية لموديول إدارة العقارات بالنماذج والواجهات والعمليات الفعلية داخل Odoo.

الغاية منها:
- تحويل المفهوم إلى كيان برمجي واضح
- معرفة أين تُخزن البيانات
- معرفة كيف تنتقل الحالة من التشغيل إلى المحاسبة
- تسهيل الصيانة والتطوير والاختبار

## 2. خريطة النماذج الأساسية

| المفهوم النظري | نموذج Odoo | الوظيفة |
|---|---|---|
| العقار | `property.property` | يمثل الأصل العقاري الأعلى |
| المبنى | `property.building` | تقسيم العقار إلى مبانٍ أو كتل |
| الدور | `property.floor` | طبقة تنظيمية داخل المبنى |

| عقد الإيجار | `property.lease` | يربط الوحدة بالمستأجر والمدة والعملة |
| جدول السداد | `property.payment.schedule` | الاستحقاقات الدورية للعقد |
| الدفعة | `property.payment` | عملية التحصيل الفعلية |

## 3. العقار `property.property`

### 3.1 دوره النظري

العقار هو الكيان المركزي الذي يمثل الأصل الثابت أو مجموعة الأصول التي تُدار كوحدة استثمارية أو تشغيلية.

### 3.2 دوره البرمجي

النموذج [`property.property`](C:\odoo\odoo\odoo\custom_addons\property_management\models\property_property.py) يحتوي على:
- الاسم
- الكود
- النوع
- المالك
- المدير
- الموقع الجغرافي
- عدد الوحدات
- نسبة الإشغال
- القيم المالية

### 3.3 الارتباط المحاسبي

تمت إضافة حقول إعدادات محاسبية على العقار نفسه، مثل:
- يومية الإيجار
- يومية الضمان
- يومية الصيانة
- حساب الذمم
- حساب إيراد الإيجار
- حساب إيراد الخدمات
- حساب التزامات الضمان
- حساب التزامات الدفعات المقدمة
- حساب مصروف الصيانة
- حساب الصندوق/التسوية

هذه الحقول تجعل كل عقار قادرًا على العمل بسياسة محاسبية مستقلة عند الحاجة.

### 3.4 الحقول المهمة

- `rent_journal_id`
- `deposit_journal_id`
- `maintenance_journal_id`
- `receivable_account_id`
- `rent_income_account_id`
- `service_income_account_id`
- `deposit_liability_account_id`
- `advance_liability_account_id`
- `maintenance_expense_account_id`
- `cash_account_id`

### 3.5 الوظائف المرتبطة

- `action_open_units`
- `action_open_leases`
- `action_open_properties`
- دوال إنشاء الحسابات واليوميات الافتراضية

### 7.1 دوره النظري

عند `action_activate`:
- تصبح الوحدة مشغولة

## 17. الترجمة إلى كود Odoo

### 17.1 الحقول المحسوبة

- الإجماليات
- الرصيد
- الإشغال
- الدخل

### 17.2 الحقول المرتبطة

الحقول المرتبطة تبسط نقل البيانات من نموذج إلى آخر:
- `company_id`
- `currency_id`
- `property_id`
- `unit_id`
- `tenant_id`

### 17.3 الحقول اليومية

الحقول اليومية والحسابات تجعل:
- الترحيل قابلًا للضبط
- المحاسبة غير hardcoded
- التوسع المستقبلي أسهل

## 18. ما الذي تغير بعد تهيئة اليمن

التهيئة لليمن تعني برمجيًا:
- دعم `YER` و`SAR`
- عدم افتراض عملة واحدة
- التعامل مع فروقات التحويل
- الفصل بين الإيراد والالتزام
- ربط التحصيل بالقيود اليومية

## 19. نقاط يجب الانتباه لها عند التطوير

- لا تضع حسابًا محاسبيًا ثابتًا داخل كود الترحيل إذا كان قابلًا للتخصيص
- لا تخلط بين deposit وrent
- لا تترك `currency_id` مربوطًا فقط بعملة الشركة
- لا تحسب `total_paid` من كل الدفعات بدون استثناء الضمانات
- لا تنشئ invoice دون يومية مناسبة

## 20. خلاصة تنفيذية

في هذا المديول:
- المفهوم النظري = كيان Odoo
- الكيان التشغيلي = سجل/عقد/دفعة
- الكيان المحاسبي = `account.move` و`account.move.line`
- الكيان الإداري = التقارير واللوحات

وبهذا تصبح البنية متماسكة:
- واضحة للمستخدم
- قابلة للمحاسبة
- قابلة للتوسع
- مناسبة لواقع السوق اليمني متعدد العملات
